How to read this report (30-second primer)

This report is generated by cc-user-autopsy, a skill that reads a user's local Claude Code usage data. It combines deterministic rule-based scoring with an LLM-written peer review.

Session
One continuous Claude Code conversation, bounded by either a fresh start or a /clear. A typical heavy user has hundreds per quarter.
Task agent / Subagent
Claude Code lets you spawn isolated child agents to run a subtask in parallel. Heavy adoption signals fluency with agentic workflows.
MCP (Model Context Protocol)
A standard for connecting Claude to external tools (Playwright, Supabase, GitHub, etc). MCP adoption rate correlates with tool breadth.
Facet
LLM-classified outcome / friction labels per session, produced by the built-in /insights command. Coverage < 100% is normal.
Interrupt recovery rate
Of sessions where the user interrupted Claude's action, what fraction still reached a "good" outcome. High = good judgment about when to stop Claude.

Peer review

Written by Claude after reading your data.

Scores above are mechanical. This section is interpretive — an attempt to identify three things you do well, three specific improvements, and one neutral observation. Every claim is meant to cite a number from your aggregate data or a specific session ID.

Three things you are doing well
  1. Strong Task agent adoption (~52% of sessions) — you're reliably fanning out work to subagents instead of doing everything in the main context. This is rare in general CC usage and shows up clearly in your data: session good-rate with Task agent (71%) is a touch higher than without (66%), and your heavy-tool sessions (>10 tool calls) are disproportionately the ones that ship commits.
  2. Your interrupt judgment is sharp — 47% of interrupted sessions still reach a good outcome. Interrupts aren't noise; they're you correctly pulling Claude back when it drifts (see session c0c9e8c3, which recovered to mostly_achieved after an early redirect).
  3. Median think-time of 78s shows deliberate prompting, not rushed — your 100-300 char prompts ship at median 14k tokens/commit vs 22k for <50 char prompts. You already use the "longer prompt pays off" lever consistently; keep it.
Three specific improvements
  1. Root-cause debugging (priority: high) — 12% of your rated sessions are iterative_refinement with buggy_code friction. That's the "v3, v4, v5…" patching pattern showing up in acme-dashboard especially. Actionable fix: add a CLAUDE.md rule — "when the same file has been edited 3+ times in one session without the error going away, stop and print a data-flow map before the next edit". This shifts you from symptom chasing to structural fixes.
  2. Tool breadth (priority: medium) — MCP usage rate is 8%, Bash/Read/Edit consume 62% of calls. You haven't pulled in tooling that exists: Playwright MCP for visual QA, Supabase MCP for schema work. Actionable fix: pick the one you'd use most and install it this week — even using it once unlocks it mentally.
  3. Afternoon friction (priority: low) — hour 15:00 has 2.8× the friction rate of 19:00 (your best hour). You produce in the afternoon but lose quality. Actionable fix: move novel problem-solving to mornings or evenings; batch routine work (docs, reviews, memory updates) to 14-16:00 when bugs are 3× likelier.
One neutral observation

Your sessions cluster in one project (acme-dashboard, ~38% of all sessions) — not a bad thing, but notice that 4 of your 5 output-token-limit hits happened there too. When a project eats half your effort, its context-hygiene overhead compounds. Worth checking whether its CLAUDE.md / directory structure is actively helping you or just incidentally carrying three months of conventions.

Methodology

What this report is — and what it is not.

Data sources

  • ~/.claude/usage-data/session-meta/*.json — auto-recorded by Claude Code.
  • ~/.claude/usage-data/facets/*.json — LLM-classified by /insights; optional but recommended.
  • ~/.claude/projects/**/*.jsonl — raw transcripts, sampled for the evidence library only.

Sampling strategy

Up to 24 sessions across 7 buckets: 5 highest-friction, 5 top-tokens, 5 most-interrupts, 4 not_achieved, 3 partially_achieved, 4 control (fully_achieved + essential), 2 user_rejected. When facets are absent, fallback is by session duration.

Caveats

Facet labels come from an LLM and may be miscategorized. Above roughly 50% facet coverage, outcome-based rules are reliable; below 30%, some dimensions return n/a. Scoring thresholds are rules of thumb, not science. The peer review depends on there being enough data to say specific things — if your data is thin, the review should be short, not padded.
